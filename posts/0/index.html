无题

发表于2026-06-28|更新于2026-06-28|AI
|字数总计:1.8k|阅读时长:5分钟|阅读量:|评论数:

一般大家使用 Claude Code 或者 Codex 编程，流程基本上就是先使用 plan 模式进行设计，然后再开启 bypass permissions on/auto review等自动审批模式开始进行实现。这叫做 Plan-and-Execute模式。

其中一般大家可能搭配superpowers里的一些skill进行设计和开发，通常在 plan 阶段使用brainstream,writing-plans,进行需求分析，使用executing-plans,tdd在执行阶段开始写代码实现。

但我觉得，对于一些复杂的需求场景，单靠一轮的plan和execute，仅完全拆分成两个阶段可能是不太合理的。很多时候plan阶段很难做到完全的可行性分析，很难确保plan出来的内容是真的可以实行的，很多时候execute阶段跑的很不顺畅，而且可能跑的甚至还越来越偏离plan，大概率都是因为plan的可行性分析没有做好评估，错误地认为execute阶段一定能完全跑通plan中的计划。所以，plan阶段一定需要人的介入，一定是要带着人的思考和评估去做，稍微大一点的工作尽量拆分成阶段性的目标，落实拆分成分阶段的计划文档，在每一阶段执行之前，人工尽量完善好该阶段的目标。

其实大模型本质上还是一个基于概率的模型，它返回的输出也只是预测出你给定的输入比较相关的一段字符串。所以对于单次的裸的LLM调用而言，你跟它谈论你比较熟悉的领域，会有有不小概率你会觉得它是在胡说八道。当你让它写一段稍微复杂一点的代码时，大概率也是编译甚至都不通过的。

ReAct 模式，也就是现在比较火的能够自主干活的 Agent，本质上是依靠 loop循环，来不断地修正反馈和引导LLM的概率输出，来让它进行持续决策输出。

Harness工程，通过给 Agent 添加 MCP，添加 Skill，设置合适的AGENTS.md约束，给它一个有价值的方向性的提示词引导，以此来给Agent设立边界和指引，让Agent能够按照我们的预期路线跑下去，而不是一直原地无效打转或者跑的越来越偏越来越远。

AI之所以如今能在写代码方面跑的通，我觉得本质上是因为写代码这件事太容易验证对错了，在自动化的情况下，试错成本极低。
在 Execute 阶段下，只要代码能够被编译器编译通过，就不存在语法问题了，只要代码能够在给定的运行环境下，被自动化测试验收通过，就可以完成我们预期想要的行为逻辑了。所以在这些严格性的约束和反馈下，天然就可以全自动约束Agent的执行路径，即使局部执行路径可能跌跌撞撞，不停试错，但是最终大概率还是能够完成到既定目标的。所以，越来越多人现在都有这么一种观点，就是代码不值钱了。

但是 Plan 阶段下，Agent 的产出是一片需求文档，计划文档，而文档先天作为一个非结构化的，不可自动化验证的，很主观的存在的内容，完全只能依靠人工的介入了，虽然文档的AI相互review可以修正一些客观的问题，但是整体的基本逻辑偏主观设计完全是体现在使用AI的人的思考和见解。

很多时候有些plan阶段设计的执行路线，如果你要是不告诉AI更合适更简单的路线，AI根本也不会告诉你可以往这个方向走才是更简单的路线，默认设计出来的路线通常并不是最优的，它很可能陷入比较复杂的方式去实现，但是有时候自己要是没有相关经验，引导给AI一个更复杂甚至错误的路线，AI依然会老实按照你给它的错误路径写入计划文档。

还有针对review代码，对于那种一次性代码可能我们觉得能用就行，但是对于正经的工程代码，一定还是需要人工介入来review代码，通常可能会发现AI堆的代码很多时候人工重构的话，还能大幅再简化，更加优雅去实现。

还有一点，人工review代码能够判断出代码是优雅简洁的，像是符合开闭原则之类的，一般都是靠传统的古法编程训练出来的一种直觉。因为人的本性就是懒，但懒又是人类的第一生产力，人写代码时候，会尽可能想方设法的少写代码，以及想方设法的重构代码，来简化后续类似功能新增的负担，尽可能少的写模板化代码，重复性代码。在这个过程中，自然而然地就训练出针对好代码的直觉。

AI写代码如果不人工review很快就形成屎山的一个原因就是，AI只知道这一次token预测输出的字符串已经可以完成当前的目标了，那就任务完成可以停下来了。AI不知疲倦，本身在已经很屎山的代码之上，还会继续堆屎山，不考虑新增的负担已经很重了，不主动考虑去重构代码，更不会想办法减少以后新增代码的负担，留下大量技术债，后续可能需要靠大量的token来还债。

由于AI的上下文是有窗口限制的，AI有注意力机制，很有可能项目到最后连AI自己都维护不下去了，这种屎山越来越大，越来越失控。所以经常有人说VibeCoding完成需求0~80%可能都很顺利，但后面就越来越难推进下去了，改了这里那里就坏掉了，改了那里这里就坏掉了。

但如今越来越多的没有经历过古法编程的人，感觉丧失了判断优雅简洁代码的直觉能力了，即使学习了设计原则设计模式的那些，没有经历过手搓也无法深入体会的到这些，主动学习的这些基本上都是死记硬背下来的知识，而不是训练出来的直觉能力。

感觉要想VibeCoding在正经项目上行得通，除非是未来模型能力真的已经上下文几乎无限大了，生成的代码每次都是最优的，到那时可能Vibe是真的可行的。就像是高级语言编译器可以把高级语言编译成汇编语言，人类不再关心底层的汇编语音这层抽象一样，AI可以把自然语言编译成计算机语言，而不再需要关心底层的计算机语言。
